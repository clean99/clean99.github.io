# Handdrawn Technical Illustrator

Use this skill to create technical illustrations that feel like a senior engineer sketched the system on a whiteboard, then cleaned it up just enough for a blog post.

The goal is not decorative art. The image should make the reader understand a mechanism faster.

## When To Use

Use this skill for:

- Runtime or compiler pipelines, such as `source -> AST -> bytecode -> JIT -> machine code`.
- Architecture layers, ownership boundaries, caches, queues, scheduling loops, or data flow.
- Performance/debugging narratives where the reader needs to see cause and effect.
- Product or workflow orientation visuals for readers who do not know the internal system.
- Blog illustrations where a Mermaid diagram would be too dry, but a photo-real image would add noise.

Do not use this skill for:

- Exact UI screenshots.
- Logos, branding assets, avatars, marketing hero images, or decorative backgrounds.
- Diagrams where every label must be pixel-perfect. Use Mermaid or a code-native diagram for those.

## Visual Style

Default style:

- Hand-drawn whiteboard or notebook diagram.
- Clean black ink lines with a few muted accent colors.
- 16:9 aspect ratio for blog hero/process images unless the article needs a tall diagram.
- High contrast, lots of whitespace, no dense paragraphs inside the image.
- Simple arrows, boxes, loops, callouts, and sticky-note style annotations.
- Labels should be short enough to remain readable at article width.
- Use real technical terms, not generic filler boxes.

Avoid:

- Tiny code blocks that readers cannot read.
- Too many stages. If there are more than 7 concepts, split the diagram.
- Fake dashboards, unreadable pseudo-UI, or invented numbers.
- Decorative gradients, glowing orbs, stock-photo backgrounds, or cute mascots.
- Brand names, internal URLs, tokens, IDs, or private infrastructure labels unless the user explicitly wants them public.

## Workflow

1. Identify the visual job in one sentence.
   - Example: "Show how V8 moves JavaScript from source to bytecode, then JIT-optimized machine code, with deopt returning to bytecode."
2. Pick the diagram shape:
   - Linear pipeline: source, transform, runtime, output.
   - Feedback loop: observe, optimize, validate, deopt/retry.
   - Layer cake: UI, host, runtime, storage, observability.
   - Ownership map: one active owner, several hidden/non-owner components.
   - Before/after: slow path vs optimized path.
3. Reduce labels before generating.
   - Put precise explanations in the article body.
   - The image should carry orientation, not every edge case.
4. Generate a raster image with the `imagegen` tool.
5. Inspect the generated image with `view_image`.
   - Confirm labels are readable.
   - Confirm arrows point in the right direction.
   - Confirm no private/internal identifiers appear.
   - Confirm the image does not invent misleading behavior.
6. Copy the chosen image into the article image folder.
   - Blog path: `source/img/<topic-slug>/<image-name>.png`
   - Markdown reference: `/img/<topic-slug>/<image-name>.png`
7. Add a caption that says what the image explains.
   - For generated blog images, include `generated by gpt-image-2` when the project convention requires it.

## Prompt Template

Use this template as the starting point for `imagegen`:

```text
A clean technical blog illustration in a hand-drawn whiteboard style showing [MECHANISM].

Diagram shape: [linear pipeline / feedback loop / layer cake / ownership map / before-after].

Include these labeled parts:
- [label 1]
- [label 2]
- [label 3]

Show these relationships:
- [A flows to B]
- [C feeds back to D]
- [E is blocked or deoptimized back to F]

Style constraints:
- high contrast black ink with muted accent colors
- readable labels, no tiny text
- no logos, no brand names, no internal URLs
- no photorealism, no decorative background
- 16:9 aspect ratio
```

## Caption Pattern

Use captions like:

```markdown
图：现代 JavaScript 在 V8 里的大致执行路径。源码先变成 AST 和 bytecode，运行时收集类型反馈，热点代码再由 TurboFan 编译成优化机器码。generated by gpt-image-2.
```

Or in English:

```markdown
Figure: Modern JavaScript execution in V8. Source code becomes AST and bytecode first; runtime type feedback lets TurboFan compile hot paths into optimized machine code. generated by gpt-image-2.
```

## Quality Bar

A good illustration passes these checks:

- A reader can explain the mechanism after looking at the image for 10 seconds.
- The image has one main idea, not five competing ideas.
- The diagram is technically honest at the level of abstraction it claims.
- The article body contains the details the image leaves out.
- The image still works if viewed on a phone.

If a generated image gets the architecture wrong, regenerate it. Do not compensate for a wrong diagram with a caption.
